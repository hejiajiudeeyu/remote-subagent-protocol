泳道

左列是当前阶段说明，右侧 3 列分别表示哪个参与方发起动作。

Buyer / Local Agent

创建请求、申请 token、投递任务、同步平台事件、拉取结果。

Platform

注册用户、目录查询、token 签发、delivery-meta、ACK 事件与可观测信息。

Seller / Remote Subagent Runtime

拉取 inbox、验证 token、ACK、执行任务、签名回包。

Stage 1

准备身份与目录上下文。这里主要与 Platform 交互。

1. 注册 Buyer

创建 buyer 用户，并把新 API key 回填到页面，供后续 `prepare`、`sync-events` 使用。

若已有 buyer key，可跳过并手工填写。

2. 读取目录与模板

先读取 catalog 锁定 `seller_id / subagent_id`，再拉取 schema bundle。

检查点

此时应已经看到 Buyer API key、选中的 subagent，以及 template schema。

Stage 2

Buyer 本地建立请求并向 Platform 申请运行上下文。

3. 创建 Buyer Request

在 Buyer Controller 里建立本地 request record，冻结 `request_id` 与 timeout 参数。

4. Prepare Request

由 Buyer Controller 代你向 Platform 申请 task token 与 delivery-meta。

检查点

此时应得到 task token、delivery address、thread hint，合同草案可用于核对输入输出结构。

Stage 3

进入实际任务链路。L0 下 seller pull 和 buyer result pull 仍然可以单独触发，用于排障。

5. Dispatch

Buyer Controller 将任务 envelope 投递到 `delivery-meta.delivery_address`。

7. Buyer Sync Events

在 Seller 已 ACK 之后，同步 Platform 里记录的 ACK 事件，把 Buyer 本地状态推进到 `ACKED`。

6. Seller Pull Inbox

驱动 Seller 运行时消费 inbox、执行任务、签名结果并回投给 Buyer。

Stage 4

结果回收与观测。可以先读 Buyer request，再看 metrics。

8. Buyer Pull Result Inbox

从 Buyer inbox 拉回 Seller result package，并更新本地 request 终态。

9. 读取 Metrics

读取 Platform 聚合摘要，确认链路事件是否进入平台可观测面。

排障提示

如果某一步失败，先看右侧日志中的请求 URL、HTTP 状态、错误码和建议的下一步。
