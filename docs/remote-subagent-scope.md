# Remote Subagent Protocol 适用范围规范（v0.1）

状态：Draft  
更新时间：2026-03-03

## 1. 规范目标

本规范用于界定哪些任务适合交给 Remote Subagent Protocol 体系承接，哪些任务不应进入当前协议参考范围，以保证：
- 交易对象可标准化（可描述、可签约、可验收）
- 买卖双方风险可控（预算、超时、权限边界明确）
- 协议实现可持续评测（成功率、延迟、合规率可量化）

## 2. 准入原则（五问门槛）

一个任务要进入 Remote Subagent Protocol 的参考范围，默认应同时满足以下 5 个条件：

1. 可合约化：输入、输出、约束可用结构化 JSON 描述。  
2. 可异步化：任务可接受异步执行，不依赖毫秒级实时交互。  
3. 可幂等化：可基于 `request_id` 重试，不产生不可控副作用。  
4. 可验收化：结果可做签名校验、Schema 校验或规则校验。  
5. 可控风险：预算上限、超时上限、失败退路、人工兜底可定义。  

任一条件不满足，应默认不纳入当前范围，或先进入实验白名单。

## 3. 典型任务分类与示例（MVP 优先）

| 分类 | 为什么适合 Remote Subagent | 典型示例 | 最小交付件 |
| :--- | :--- | :--- | :--- |
| 文档与格式工程 | 本地一次跑通成本高，格式细节依赖专用工具链与经验 | `Markdown -> Word(.docx)`；`PDF -> Markdown`（表格/公式/标题层级还原） | 转换产物 + 质量报告（还原率、失败页、警告项） |
| 内容渠道化生产 | 同源内容到多平台素材的拆分、改写、生成通常链路较长 | `Markdown -> 小红书图文卡片`（封面图、分卡文案、标签建议） | 图片包 + 卡片文案 + 发布元数据 |
| 专业领域排障与诊断 | 需要领域规则、经验库和排障流程，不是单次 prompt 即可稳定完成 | 分子实验问题排查（失败原因排序、复现实验建议、风险提示） | 诊断结论 + 证据链 + 下一步操作清单 |
| 基础设施密集型任务 | 需要专有模型、私有服务、GPU/HPC 或复杂依赖环境 | 高成本 OCR/解析流水线、专用模型推理、私有知识库编排 | 结果包 + 运行摘要（耗时、资源、关键日志） |
| 标准化批处理任务 | 任务可批量化、指标化，适合做稳定 SLA 和横向比较 | 批量字段抽取、文本分类打标、质量筛查 | 结构化结果 + 置信度 + 异常清单 |

## 4. 当前不适合（红线）

以下任务不应作为 Remote Subagent Protocol 常规收录对象：

1. 强实时交互任务（例如低延迟双向协作、实时控制闭环）。  
2. 必须访问买家本地运行时、内网资源或本地密钥的任务。  
3. 高风险不可逆动作（直接支付、生产写操作、法律/医疗最终裁决）。  
4. 无法客观验收、仅靠主观偏好判断结果质量的任务。  

## 5. 远景范围（v0.2+ 探索）

在机制完善后，Remote Subagent 可逐步扩展到以下方向：

1. 远程真实世界实验：涉及远程设备、实验台、线下采样与结果回传。  
2. 人参与任务：Human-in-the-loop 审核、复核、执行节点。  
3. 高门槛基础设施任务：专用仪器、企业私有系统、长时作业流水线。  
4. 经验积累型任务：依赖长期案例沉淀与专家策略复用。  

## 6. 上架治理建议

1. 参考实现中的目录收录，应附带任务分类、风险等级、验收规则与失败退路。  
2. 协议参考实现的审核，应使用 `docs/checklists/subagent-admission-checklist.md` 统一评估。  
3. 对于“远景范围”任务，默认进入白名单或灰度，不直接全量开放。
